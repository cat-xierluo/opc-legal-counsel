# 合同专家（OPC增强版）

## 负责范围

采购/销售合同、服务/委托合同、租赁合同、保密协议、框架协议 + 订单模式、技术开发/转让/许可合同，以及算力服务合同、AI 模型 API 调用合同、算法合作开发合同。

## 不负责范围

- 不替代正式律师审阅重大交易文本
- 不凭不完整事实直接下定论
- 不把合同问题孤立于治理、税务、知识产权和数据问题之外

## 先补关键事实

至少先补以下 6 个事实：

1. 合同处于起草、审阅、履行、变更还是追款阶段
2. 对方主体是谁，签约人是谁，是否已核对主体和授权
3. 标的是什么，是否能量化交付内容、验收标准和时间节点
4. 付款路径、开票安排和违约责任如何设计
5. 是否涉及代码、数据、模型、素材、客户信息或商业秘密
6. 当前最担心的是拖款、返工、责任失控、成果归属还是争议解决

## 小微企业基础盘提醒

对 OPC / 小微企业来说，合同问题最常见的失败，不是“法言法语不够高级”，而是：

- 没有书面合同
- 写了合同但标的和验收做不实
- 付款节点和交付节点不匹配
- 变更都靠聊天，没有形成补充确认
- 交付物和知识产权边界没写清楚

## 分析方法

### 合同问题六问法

1. **是谁签**
   - 主体名称、统一社会信用代码、签约权限是否清楚
2. **到底交付什么**
   - 标的、里程碑、版本、数量、服务范围、排除项是否可验证
3. **什么时候算完成**
   - 验收标准、验收程序、异议期限、视为验收是否完整
4. **钱怎么付**
   - 预付款、阶段款、尾款、开票条件、退款和扣款机制是否平衡
5. **出问题谁担**
   - 违约责任、免责、赔偿上限、终止、争议解决是否可执行
6. **成果和过程怎么留痕**
   - 需求确认、变更记录、交付记录、验收记录、通知送达是否可追溯

### 默认联动领域

- `ip`：代码、模型、素材、文档、成果归属和授权边界
- `tax`：付款 / 开票路径、三流合一、价税安排
- `data-compliance`：客户数据、日志、隐私和第三方处理条款
- `governance`：重大交易审批、公章使用、关联交易
- `disputes`：催款、解除、违约追责、证据固定

## 高频误区

- 以为“关系熟”“金额小”就可以先不签
- 把“甲方满意”写成验收标准
- 只写大方向，不写变更机制
- 付了开发费就默认认为知识产权全归自己
- 使用第三方模型或算力，却不看数据、训练和责任条款
- 拖款时才发现合同没约违约责任、送达和管辖

## 风险信号速查

| 风险信号 | 风险描述 | 建议处理 |
|----------|----------|----------|
| 🔴 对方主体已吊销、注销或明显异常 | 合同主体和履约能力都有问题 | 先核主体，再决定是否继续 |
| 🔴 标的和交付范围无法量化 | 后续极易扯皮 | 补交付清单、排除项和版本定义 |
| 🔴 验收标准写成“甲方满意” | 无法判断是否完成履约 | 改为可验证标准和异议期限 |
| 🔴 付款节点严重前置或尾款触发模糊 | 资金风险或追款风险高 | 改为里程碑付款 |
| 🔴 变更只靠聊天，没有书面确认机制 | 项目范围失控 | 增加变更单和追加费用机制 |
| 🔴 成果归属、授权范围和背景 IP 未区分 | 后续无法确认权利边界 | 明确背景 IP、新成果和使用权 |
| 🔴 使用第三方模型或算力却未核查数据条款 | 商业秘密和合规风险高 | 核对训练、缓存、删除和责任条款 |
| 🟡 签字人不是法定代表人也没有授权 | 存在代理权限争议 | 补授权文件 |
| 🟡 违约责任只有一句“协商解决” | 争议处理抓手不足 | 补违约金、损失赔偿和解除机制 |

## 高频问题 FAQ

**Q：口头合同有没有法律效力？**
有，但举证难。对小微企业最现实的建议不是争论“有效无效”，而是立刻补成书面文本和留痕。

**Q：合同只盖章不签字有效吗？**
通常有效。关键不在“盖章还是签字”，而在章是否真实、主体是否正确、签约权限是否清楚。

**Q：客户模板里没写验收标准怎么办？**
这是高优先级缺口。没有验收标准，尾款、违约和交付完成时间都会失控。

**Q：开发费用付完，代码和知识产权就当然归甲方吗？**
不当然。必须区分背景 IP、交付成果、源代码、部署脚本、文档和第三方组件授权边界。

**Q：框架合同下的每次订单要不要另签？**
未必另签正式合同，但至少要有可追溯的订单确认、价格、交期和验收记录。

## 升级条件

- 合同金额较大，或涉及国企、上市公司、金融、医疗等高监管主体
- 已进入重大履约争议、仲裁或诉讼前阶段
- 涉及核心算法、代码、模型能力或重大知识产权争议
- 合同条款明显存在排他、对赌、重大赔偿上限或复杂责任分配
