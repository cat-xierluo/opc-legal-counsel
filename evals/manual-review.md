# 人工复核说明

本文件用于人工复核 `evals/evals.json` 中的重点样本。它不是新的法律知识库，而是判断回答质量是否稳定的评分协议。

## 总体评分

每条样本按 10 分制评分。

| 维度 | 分值 | 评分要点 |
|------|------|----------|
| 输出结构 | 2 | 是否保留结论、领域、事实缺口、风险、动作、升级边界、依据这些核心要素 |
| 路由与下一跳 | 2 | 是否命中主领域、联动领域、专项模块、行业 overlay、样稿或地方覆盖层 |
| 事实识别 | 2 | 是否补关键事实或写明假设，是否识别会改变结论的事实 |
| 风险与动作 | 2 | 是否把风险落成 24 小时、7 天、30 天动作，而不是只讲概念 |
| 升级边界 | 2 | 是否在应升级时升级，并说明升级前的止损材料和准备清单 |

## 通过标准

- `8-10 分`：通过，可作为合格输出
- `6-7 分`：可用但需要修订，记录问题后进入下一轮优化
- `0-5 分`：不通过，应回到 `SKILL.md` 或对应 reference 调整

## 硬失败条件

只要出现以下任一情况，即使分数较高，也应判为不通过：

- 把地方规则说成全国通用规则
- 把跨领域问题拆成互不关联的分科回答，没有综合判断
- 在事实缺失时替用户编造关键事实
- 高风险事件没有触发升级边界
- 涉及税率、备案名单、平台规则、监管公告等易变事项时，未提示核验
- 给出明显会扩大风险的动作，例如数据泄露后建议删日志、监管问询时随意承认责任

## 重点样本复核

### foundation-02-first-client-contract

**复核重点：合同审查是否从条款缺口落到可执行修改。**

必须命中：

- 指出验收标准、变更机制、尾款触发条件是高优先级问题
- 识别 `contracts + ip + governance`，而不是只讲付款
- 调用或参考 `assets/contract-clauses.md` 与 `assets/output-samples/contract-review-report.md`
- 提醒需求确认、版本交付、验收和上线留痕
- 不把“项目完成”当成足够明确的付款条件

硬失败：

- 只说“建议协商补充条款”，没有具体条款方向
- 没有处理知识产权归属和交付物边界

### foundation-03-private-account-collection

**复核重点：公私混同是否被识别为红线，而不是普通流程问题。**

必须命中：

- 明确私户收公款是 OPC 红线风险
- 要求立即停止继续用个人账户收公司业务款
- 区分治理风险、税务风险和争议举证风险
- 给出历史流水、合同、发票、记账路径的补救清单
- 金额较大、跨期或票据混乱时，提示会计 / 税务顾问介入

硬失败：

- 说“后续转回公司即可解决”
- 没有提示公私分离和股东责任风险

### reinforcement-03-ai-launch-checklist

**复核重点：AI 上线检查是否覆盖专项合规、数据、宣传和 IP。**

必须命中：

- 识别 `ai-compliance + data-compliance + regulatory + ip`
- 调用或参考 `assets/output-samples/ai-launch-report.md`
- 覆盖用户协议、隐私政策、模型来源、公示、投诉入口和日志留痕
- 提示生成内容标识和营销表述审查
- 对备案、公示名单、监管公告等易变事项提示核验

硬失败：

- 只回答隐私政策，不处理 AI 标识 / 公示
- 把内部工具和公开 C 端产品按同一强度处理

### reinforcement-05-data-breach-public-incident

**复核重点：数据事件是否先止损、封存、升级，而不是客服化处理。**

必须命中：

- 立即停止扩散、回收权限、封存日志
- 明确外部律师、安全或合规团队应介入
- 不在事实未清前随意承认责任或发布失真说明
- 要求整理影响范围、时间线、系统日志、通知和处置记录
- 评估是否通知个人、客户、合作方或监管部门，并提示核验法源

硬失败：

- 建议删除日志或先压热度不留痕
- 没有触发升级边界

### reinforcement-06-local-overlay-qingdao

**复核重点：地方覆盖层是否正确叠加，而不是污染全国基线。**

必须命中：

- 先给全国通用规则，再叠加青岛地方材料
- 明确地方材料只作为青岛地方实践或指引参考
- 优先读取 `references/local-policies/opc/05-青岛指引-结构化摘录.md`
- 不直接依赖原始 OCR 全文
- 说明哪些结论会因地域变化而变化

硬失败：

- 把青岛材料说成全国规则
- 不区分全国基线和地方覆盖层

### reinforcement-07-growth-advisor-equity

**复核重点：顾问股问题是否命中成长阶段模块，并拆清法律工具。**

必须命中：

- 命中 `growth-financing + governance + contracts + tax`
- 区分顾问股、收益分成、期权、现股和附条件持股
- 识别兼职顾问直接给现股的控制权和退出风险
- 补服务期限、具体对价、归属条件、回收机制和税务节点
- 提醒口头承诺会影响后续股权结构表和融资

硬失败：

- 直接回答“给 3% 可以 / 不可以”，没有拆法律工具
- 没有处理归属和退出机制

### reinforcement-12-industry-ai-saas-beta

**复核重点：AI SaaS beta 是否被识别为上线前合规场景。**

必须命中：

- 命中 `ai-compliance + data-compliance + contracts + regulatory + industry/ai-saas`
- 说明 beta 不是免合规阶段
- 覆盖试用协议、客户数据边界、模型来源、公示、日志和删除机制
- 识别录音和聊天记录可能涉及个人信息和敏感信息
- 如依赖最新监管公告，提示官方核验

硬失败：

- 只从产品试用或合同角度回答，不处理数据和 AI 专项义务
- 认为只有 10 家客户就不需要合规

### reinforcement-14-employment-contractor-core-dev

**复核重点：外包工程师争议是否同时处理用工、合同、IP 和争议。**

必须命中：

- 命中 `employment + contracts + ip + disputes`
- 立即提示回收或分级调整代码仓库、密钥、后台和客户资料权限
- 补问合同、提交记录、付款记录、代码贡献和权属条款
- 同时分析劳动关系风险和外包成果归属
- 核心代码权属已经争议时提示律师介入

硬失败：

- 只按劳动关系或只按外包合同单线回答
- 没有提醒权限回收和证据固定

### reinforcement-16-regulatory-market-inquiry

**复核重点：监管问询是否按正式监管场景处理。**

必须命中：

- 命中 `regulatory + ai-compliance + disputes`
- 提醒保存原页面、修改记录、投放记录和整改说明
- 识别 AI 教育产品宣传属于高风险组合
- 提醒核对通知范围，不随意扩大承认责任
- 已立案或收到处罚告知时提示律师介入

硬失败：

- 只说“已经改掉就可以”
- 建议用户未核实事实就承认违法

### reinforcement-18-cross-domain-user-complaint-ai-ecommerce

**复核重点：跨领域问题是否形成综合判断，而不是分科罗列。**

必须命中：

- 命中 `contracts + regulatory + ip + ai-compliance + disputes`
- 同时叠加 `industry/ecommerce` 和 `industry/agency-outsourcing`
- 先看合同责任边界、客户确认记录、素材来源和平台处罚依据
- 区分客户自身商品事实、AI 生成内容、客户审核发布和平台规则
- 提醒固定生成记录、客户确认、发布记录、投诉和处罚材料

硬失败：

- 直接把责任全部推给客户或代运营方
- 只讲广告法或只讲合同，不做综合判断

## 复核记录建议格式

人工复核时建议按以下格式记录：

```markdown
### eval-id

- 分数：X/10
- 结论：通过 / 需修订 / 不通过
- 主要问题：
- 缺失要点：
- 是否触发硬失败：
- 建议调整：
```
