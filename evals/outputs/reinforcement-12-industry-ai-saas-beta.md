# reinforcement-12-industry-ai-saas-beta

## 用户问题

"我们做一个面向企业的 AI 质检 SaaS，客户会上传通话录音和客服聊天记录，我们打算先开放 beta 给 10 家客户试用。现在合同、页面和流程上最少要补哪些东西？"

---

## 1. 一句话结论

通话录音和聊天记录涉及个人信息甚至敏感个人信息，开放 beta 前最少要补齐三类东西：用户协议和隐私政策（写清数据怎么处理）、客户试用协议（锁定数据边界和免责范围）、产品页面的 AI 标识和投诉入口——缺任何一个都可能踩到合规红线。

## 2. 涉及领域

- **主领域**：ai-compliance（AI 产品上线标识、公示、模型核验）、data-compliance（个人信息、敏感信息处理、隐私政策）
- **联动领域**：contracts（客户试用协议、SLA、责任边界）
- **核验领域**：ip（输出成果归属）、regulatory（宣传表述）

## 3. 关键事实缺口 / 当前假设

| 缺口 | 当前假设 | 影响说明 |
|------|----------|----------|
| 使用自研模型还是第三方模型/API | 假设调用第三方已备案模型的 API | 需要核验上游模型备案状态，并在产品页公示 |
| 通话录音是否包含语音生物识别信息 | 假设可能包含声纹等生物特征信息 | 如果涉及，属于敏感个人信息，需要单独同意 |
| 聊天记录是否包含客户（终端用户）的个人信息 | 假设包含客户姓名、联系方式等 | 需要明确企业客户是否有权上传其终端用户数据 |
| beta 是否收费 | 假设免费试用 | 免费不影响合规义务，但合同条款设计可以更灵活 |
| 是否涉及金融、医疗等高监管行业客户 | 假设暂不涉及 | 如果涉及，数据保护要求更高 |
| 已有隐私政策和用户协议 | 假设尚无或不完善 | 必须在 beta 前补齐 |

## 4. 红线风险与风险等级

| 风险 | 等级 | 说明 |
|------|------|------|
| 处理通话录音和聊天记录未告知个人、未获合法依据 | 🔴 高 | 通话录音通常包含个人信息，甚至可能包含敏感个人信息；没有告知和同意就是违法处理 |
| 上游模型未核验备案/登记状态 | 🔴 高 | 使用未备案模型对外提供服务，可能被认定为违规 |
| 没有隐私政策和用户协议 | 🔴 高 | 《个人信息保护法》要求处理个人信息前必须告知，且告知必须真实、准确、完整 |
| 没有投诉举报和下架机制 | 🔴 高 | 生成式 AI 服务应具备投诉处置能力 |
| beta 期间数据泄露或被模型训练 | 🔴 高 | 客户上传的商业录音和聊天记录如果泄露或被用于训练，后果严重 |
| 产品页未公示 AI 模型信息 | 🟡 中 | 网信办公告要求公示模型名称和备案号 |
| 没有明确的数据留存和删除机制 | 🟡 中 | beta 结束后客户数据怎么处理需提前约定 |

## 5. 现在就做什么（24 小时内）

1. **核验上游模型备案状态**：登录国家网信办官网，确认你调用的模型是否在已备案/登记名单中，记录模型名称和备案号。
2. **列出数据处理清单**：明确——
   - 客户上传什么数据（录音文件、聊天记录文本、元数据）
   - 数据怎么处理（质检分析、转写、评分、报告生成）
   - 数据存在哪里（你的服务器、云存储、模型 API 传递）
   - 数据保留多久
   - 谁能访问
3. **检查第三方模型服务商条款**：确认 API 调用时客户数据是否会被缓存、训练或存储，是否有企业版数据处理协议（DPA）可签。
4. **暂停所有尚未签署试用协议的客户试用**，直到基础合规文件就位。

## 6. 接下来怎么做（7 天 / 30 天）

### 7 天内：beta 前必须补齐的"最小可用合规包"

**一、合同层面**

1. **起草 Beta 试用协议**，至少包含：
   - 试用范围、功能边界和期限（建议明确标注"Beta 测试版，非正式商用版"）
   - 客户保证其上传数据已获得终端用户同意（数据合法性由客户负责）
   - 数据处理说明：数据如何被处理、存储多久、是否会传输给第三方模型
   - 免责条款：beta 版不保证服务质量、不承诺 SLA、不对质检结论承担商业决策责任
   - 输出成果（质检报告等）的归属和使用范围
   - beta 结束后数据的删除或迁移安排
   - 保密义务双向约束

**二、页面层面**

2. **产品页/功能页补充以下内容**：
   - 公示所使用的 AI 模型名称和备案号/上线编号
   - AI 生成内容的标识（至少在功能页标注"本服务使用 AI 技术进行质检分析"）
   - 风险提示：质检结果仅供参考，不构成对客服质量的最终评判
   - 投诉举报入口

**三、流程层面**

3. **补齐隐私政策/数据处理说明**，至少包含：
   - 谁是数据处理者（你和客户的关系）
   - 处理什么数据、为什么处理、处理方式
   - 数据保存期限
   - 用户权利（查询、更正、删除）的行使方式
   - 第三方模型服务商的信息和处理方式
   - 联系方式

4. **设置数据保留和删除机制**：beta 期间设定合理的数据保留期（如 30 天），beta 结束后按约定删除或归还客户数据。

### 30 天内

1. **根据 beta 反馈迭代合规体系**：
   - 补充正式版的用户协议和服务条款
   - 建立内容审核和投诉处置的标准流程
   - 评估是否需要生成内容的文件级标识（导出报告是否需要标注 AI 生成）
2. **建立数据安全基线**：
   - 访问权限最小化（谁能看客户数据）
   - 操作日志留痕
   - 加密存储和传输
3. **准备正式商用版的合规清单**：
   - 对照《生成式人工智能服务管理暂行办法》逐项检查
   - 评估是否需要做算法备案或安全评估
   - 准备商用版合同模板（含 SLA、赔偿上限、数据 DPA）

## 7. 何时必须升级给外部律师 / 会计师 / 专业机构

- 客户数据发生泄露或被不当使用
- 收到网信、市场监管等部门的问询或整改通知
- 客户涉及金融、医疗、教育等高监管行业，对数据合规有专项要求
- 准备从 beta 转为正式商用，需要完整的合规体系搭建
- 客户要求签署正式的数据处理协议（DPA）或安全评估报告

## 8. 法律依据与适用层级

| 依据 | 层级 | 适用说明 |
|------|------|----------|
| 《个人信息保护法》第6-7条 | 全国法律 | 个人信息处理应遵循最小必要、公开透明原则 |
| 《个人信息保护法》第13-14条 | 全国法律 | 处理个人信息需有合法依据；敏感个人信息需取得单独同意 |
| 《个人信息保护法》第17条 | 全国法律 | 处理个人信息前应以显著方式、清晰易懂的语言告知 |
| 《个人信息保护法》第47条 | 全国法律 | 个人信息处理目的已实现或不再必要时，应主动删除 |
| 《数据安全法》第27-31条 | 全国法律 | 数据安全保护义务、风险评估和报告要求 |
| 《生成式人工智能服务管理暂行办法》 | 部门规章 | 生成式 AI 服务的备案、公示、标识、投诉处置要求 |
| 《人工智能生成合成内容标识办法》（2025-09-01 实施） | 部门规章 | AI 生成合成内容的标识义务 |
| GB 45438-2025 | 国家标准 | AI 生成合成内容标识方法，2025-09-01 起实施 |
| 国家网信办公告（2025-04-08 等） | 监管公告 | 已备案生成式 AI 服务信息公示要求 |

**注意**：通话录音可能涉及《个人信息保护法》第28条规定的敏感个人信息（声纹、生物识别信息），如果涉及，合规要求更高。建议在 beta 前由执业律师或合规顾问复核数据处理方案。
